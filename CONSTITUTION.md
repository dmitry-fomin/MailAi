# Конституция проекта: MailAi

Этот документ фиксирует философию проекта. CLAUDE.md отвечает на вопрос «как делаем», CONSTITUTION.md — «почему так».

## Архитектурные принципы

- **Модульный монолит на SPM-пакетах.** Один app-target, бизнес-логика вынесена в локальные Swift Package-модули с явными границами и `public`-API.
  - Предварительный набор модулей: `Core` (модели, протоколы, утилиты), `MailTransport` (IMAP, Exchange), `Storage` (SQLite/GRDB, только метаданные), `AI` (OpenRouter-клиент, промпты, суммаризация), `App` (SwiftUI + AppKit, view models).
- **Многооконность по модели IDE.** Каждый почтовый аккаунт открывается в своём окне приложения — как проекты в Xcode/JetBrains. У каждого окна свой изолированный state, свой кеш в памяти, свой жизненный цикл сетевых коннекторов.
- **Протоколы как границы.** IMAP/Exchange/AI-провайдеры скрыты за протоколами (`MailTransport`, `AIProvider`). Замена реализации или добавление нового провайдера не должна трогать UI и бизнес-логику.
- **MVVM + сервисы.** Views → ViewModels (`@MainActor`, `ObservableObject`/`@Observable`) → Services/Repositories (акторы). Никаких циклических зависимостей, никакой бизнес-логики во views.
- **Unidirectional data flow.** State — единый источник правды, изменения идут через интенты/события, а не через двустороннюю мутацию.

## Приоритеты

В порядке убывания — когда конфликтуют, выигрывает то, что выше:

1. **Минимум памяти и ресурсов.** Это профильное требование продукта: тела писем не задерживаются в памяти дольше необходимого, крупные вложения стримятся, кеш ограничен по размеру и времени жизни.
2. **Скорость UI.** Любое действие пользователя даёт отклик в пределах кадра (≤16 мс). Сеть и тяжёлые вычисления — только вне главного актёра, список писем скроллится без заиканий даже на 100 000 писем в ящике.
3. **Качество кода.** Strict Concurrency, покрытие тестами критичных путей, SwiftLint без warnings. Лучше сделать медленнее и правильно, чем быстро и с техдолгом.
4. **Гибкость.** Возможность подменить транспорт (IMAP → Exchange → JMAP) и AI-провайдера без переписывания UI. Достигается за счёт протоколов, а не ради самой абстракции.
5. **Скорость разработки.** Важна, но не ценой приоритетов выше. MVP срезается по функциональности (меньше фич), а не по качеству или экономии ресурсов.

## Ограничения и политики

### Приватность и данные

- Тела писем, вложения, фрагменты текста **никогда** не записываются на диск — ни в кеш, ни в БД, ни в логи, ни в крашдампы.
- В локальной БД хранятся только метаданные: message-id, заголовки, флаги, даты, размеры, UID'ы IMAP, хеши для дедупликации.
- Отправка текстов писем в OpenRouter — только по явному действию пользователя (суммаризация, анализ конкретной переписки). По умолчанию письма наружу не уходят.
- В логи не попадают: содержимое писем, адреса отправителей/получателей, токены, пароли, API-ключи. Диагностические логи содержат только счётчики, идентификаторы операций и типы ошибок.

### Секреты

- Пароли IMAP/Exchange, OAuth-токены, OpenRouter API-ключи хранятся только в Keychain (scope — приложение, не shared).
- Для доступа к Keychain используется один централизованный сервис, обёрнутый в протокол.

### Функциональность MVP

- **Поддерживается**: чтение, удаление (в т.ч. массовое по запросу), AI-анализ, AI-суммаризация, отправка писем.
- **Многоаккаунтность**: с самого начала, модель «окно = аккаунт» (как проекты в IDE). Переключение контекста между окнами, а не внутри одного окна.
- **Офлайн**: nice-to-have, не блокер MVP. Минимум — список писем и метаданные доступны без сети за счёт локального индекса.

### Распространение

- Подписывание и нотаризация пока не требуются. Сборка для локального запуска.
- Когда дойдёт до дистрибуции — отдельная задача с настройкой signing/notarization.

### Производительность

- Главный поток не блокируется сетью и I/O. Ни при каких условиях.
- Список писем на 100 000+ элементов скроллится плавно (lazy-rendering, виртуализация).
- Суммаризация переписки и анализ — фоновые задачи с прогрессом и возможностью отмены.

## Code Style

- **SwiftLint**: строгая конфигурация, warnings = errors в CI.
- **Форматирование**: swift-format (или встроенный Xcode formatter) — единый стиль в команде.
- **Именование**: Swift API Design Guidelines. `camelCase` для свойств/функций, `PascalCase` для типов, без венгерской нотации, без префиксов вроде `MA`.
- **Файлы**: один основной тип на файл, имя файла = имя типа.
- **Async**: `async/await` и структурированная конкурентность. Никаких `DispatchQueue.main.async` в новом коде — используем `@MainActor`.
- **Ошибки**: типизированные `enum Error`, без `throws` как «swallow all». Пользовательские сообщения отделены от технических.
- **Комментарии**: только «почему», а не «что». Хорошее имя заменяет комментарий.

## Запрещено

- Писать тела писем и их фрагменты в любое персистентное хранилище.
- Логировать содержимое писем, адреса, секреты.
- Сторонние UI-библиотеки (SnapKit, Introspect, alternative SwiftUI-компоненты).
- Completion handlers и `DispatchQueue` в новом коде.
- Бизнес-логика во views и в моделях данных.
- Синглтоны для сервисов (кроме технических утилит). Зависимости передаются явно.
- «Божественные» модули — если пакет разросся, разделяем по доменной границе.
- Абстракции «про запас» без реального второго случая использования.
