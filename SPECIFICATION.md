# Спецификация MVP: IMAP-аккаунт + 3-колоночный UI + многооконность

## Проблема

У пользователя нет нативного macOS-клиента, который одновременно:
- выглядит и ведёт себя как стандартный Mail.app (привычный 3-колоночный layout),
- поддерживает многооконность «окно = аккаунт» (как проекты в IDE),
- не хранит тела писем на диске и экономит память.

Эта фича закладывает визуальный и архитектурный каркас, на который дальше сядут AI-функции (суммаризация, bulk-delete), StatusBar, Search, Notifications.

## Пользователь

Владелец 1–3 почтовых ящиков на macOS, который:
- держит Mail.app/Spark/Thunderbird открытым весь рабочий день,
- ценит нативный внешний вид и горячие клавиши,
- не доверяет облачным AI-клиентам и хочет контроля.

## Скоуп фичи

### Входит

1. **Onboarding**: экран подключения первого IMAP-аккаунта (server, port, TLS, email, пароль — в Keychain).
2. **Многооконность**: каждый аккаунт — отдельное окно. «File → New Account Window…» открывает новое окно под другой аккаунт. State restoration набора окон при перезапуске.
3. **3-колоночный layout** в каждом окне:
   - **Sidebar**: секции «Избранное», «Смарт-ящики» (заглушки на MVP), «На моём Mac», `<email>` с папками.
   - **Список писем**: заголовок mailbox-а + счётчик, строки писем (отправитель, время, тема, превью-строка, индикаторы вложения/ответа), выделение активного.
   - **Reader**: toolbar (ответить / переслать / архив / удалить / флажок / поиск), header письма (аватар, from, to, дата), тело письма (стримом).
4. **Моковые данные**: при запуске без подключённого аккаунта UI работает на in-memory mock-датасете (100–200 писем, папки, тред с 2–3 сообщениями). Используется для разработки UI и скриншотов.
5. **Подключение реального IMAP** (отдельный экран): после успешного login пишем метаданные в `Storage`, переключаем UI с mock на реальные данные.
6. **Список писем INBOX**: первые 200 последних писем по UID, пагинация по скроллу.
7. **Открытие письма**: при клике в списке — стриминговая загрузка тела, рендер в reader. Тело живёт только пока письмо открыто.
8. **Навигация с клавиатуры**: ↑/↓ по списку, ⌘1/⌘2 — фокус sidebar/list, Space — прокрутка reader, ⌘R — обновить.

8. **UI-каркас для будущего AI-pack** (заглушки, без логики): секция sidebar «Отфильтрованные» с пустыми папками `Важное (0)` и `Неважно (0)`, зарезервированный (collapsed) слот прогресс-бара под header'ом списка, зарезервированный слот иконки фоновой синхронизации в toolbar'е reader'а, пункт-placeholder «AI-классификация — недоступно» в Settings. Подробности — в [docs/superpowers/specs/2026-04-17-ai-pack-design.md](docs/superpowers/specs/2026-04-17-ai-pack-design.md).

### Не входит (откладывается)

- AI-функции (классификация Важное/Неважно, суммаризация, bulk-delete) — отдельный AI-pack, спека есть.
- Exchange/Graph — после IMAP.
- Отправка писем — после чтения.
- Поиск (FTS5 + server-side) — отдельная фича.
- StatusBar-иконка — отдельная фича.
- Уведомления, смарт-ящики, флажки (на MVP — только UI-заглушки).
- Удаление писем (в т.ч. обычное) — следующая фича после MVP.
- Вложения: только индикатор в списке, полноценный просмотр/сохранение — позже.

## User Stories

1. Как новый пользователь, я добавляю IMAP-ящик через форму и через 2 клика вижу свои письма.
2. Как пользователь с двумя ящиками, я открываю второй через File → New Account Window… и переключаюсь между ящиками через Window-меню / ⌘`.
3. Как дизайнер-фронтендер, я могу запустить приложение без реального ящика и увидеть полноценный UI на моковых данных — для прототипирования и скриншотов.
4. Как пользователь, я кликаю по письму и вижу его текст через ≤500 мс на канале 10 Мбит/с.
5. Как пользователь Mail.app, я ожидаю привычных клавиатурных сокращений и layout'а.

## Сценарии

### Happy Path: первое подключение

1. Запуск без аккаунтов → открывается **Welcome window** с кнопкой «Добавить аккаунт».
2. Форма: email, пароль, сервер (IMAP host/port/TLS), опционально SMTP-поля (серые, disabled — отправки нет в MVP).
3. Нажатие «Продолжить» → проверка коннекта (LOGIN + LIST).
4. Успех → пароль уходит в Keychain, аккаунт — в `Storage`, Welcome закрывается, открывается окно-аккаунт с загруженным INBOX.

### Happy Path: просмотр письма

1. Окно-аккаунт открыто, в списке выбрано письмо.
2. Reader показывает spinner → начинается `FETCH BODY[]` стримом.
3. По мере поступления чанков — рендерим текст/HTML инкрементально.
4. Закрытие письма (выбор другого / ⌘W / сворачивание окна) → `MessageBody` `= nil`, память освобождается.

### Happy Path: многооконность

1. В открытом окне-аккаунте пользователь жмёт **File → New Account Window…**.
2. Появляется picker: список подключённых аккаунтов + «Добавить новый…».
3. Выбор существующего → открывается новое окно под этот аккаунт с собственной сессией.
4. Перезапуск приложения → восстанавливаются все окна, которые были открыты (без содержимого писем — только layout и выбранные mailbox'ы).

### Happy Path: моковые данные

1. Запуск с флагом `--mock` (или пункт меню «Developer → Mock Data») → открывается окно с фиктивным аккаунтом.
2. Видны папки, письма, треды, можно кликать и читать — всё из in-memory фикстур.
3. Никаких сетевых запросов.

### Edge Cases

- **Неверный пароль** → forma подключения показывает ошибку, секрет в Keychain не записывается.
- **Нет интернета** → список показывается из `Storage` (то, что успели засинкать), reader показывает сообщение «Нет связи, тело письма недоступно».
- **UIDVALIDITY изменился** → локальный индекс mailbox'а инвалидируется, показываем индикатор синка, перезабираем headers.
- **Пустой INBOX** → empty state с иллюстрацией и подсказкой.
- **Mailbox с 100k+ писем** → виртуализованный список (AppKit-таблица через `NSViewRepresentable`), первые 200 в памяти, остальные по скроллу.
- **Огромное письмо (>10 MB)** → стриминг, прогресс-бар в reader.
- **Пользователь закрыл все окна** → приложение остаётся в Dock (и будет доступно через StatusBar в следующих фичах).
- **Два окна одного аккаунта** — запрещаем (открытие того же аккаунта фокусирует существующее окно).

## UI-требования (по референсу)

3-колоночный layout с `NavigationSplitView` (или AppKit-аналог, если SwiftUI просаживает виртуализацию).

**Sidebar:**
- Секции: «Избранное», «Смарт-ящики», «На моём Mac», `<email-аккаунта>`.
- Для каждой папки: иконка, имя, опциональный счётчик непрочитанных справа.
- Поддержка раскрытия/сворачивания вложенных папок.
- Активная папка — выделена системным акцентом.

**Список писем (middle):**
- Хедер: название mailbox'а («Входящие — Золото»), ниже — «N писем, M непрочитанных», справа — кнопки фильтра и сортировки (заглушки в MVP).
- Строка письма: отправитель (bold), время (справа, кратко — «18:05» / «Вчера» / «28 авг»), тема, 2-строчный превью (первые ~150 символов темы + первой строки — **внимание**: превью берётся из IMAP `BODYSTRUCTURE` preview, если сервер даёт; иначе генерируем из `ENVELOPE.subject`, без загрузки тела).
- Индикаторы: скрепка (вложение), стрелка reply (ответ в треде), счётчик сообщений треда.
- Непрочитанные — с синим пойнтом слева.
- Выделенное письмо — акцентный фон.

**Reader (right):**
- Toolbar иконок: новое письмо, ответить, ответить всем, переслать, архив (заглушка), удалить (заглушка в MVP), флажок (заглушка), поиск.
- Header письма: аватар-заглушка (инициалы), имя отправителя, дата, справа иконка скрепки. Ниже — «Кому: ...» + «Подробнее».
- Тело: рендер plain text или sanitised HTML (без автозагрузки внешних ресурсов).
- Для треда — список писем треда друг за другом (как на скриншоте).

**Темы:**
- Light и Dark — обе поддерживаются (скриншот референса — Dark).
- Нет собственных цветовых констант — только системные семантические.

## Критерии приёмки

Финальный прогон C6 (2026-04-18). Галочки проставлены по факту
покрытия кодом и smoke-тестами в `Scripts/smoke.sh`.

- [x] Приложение запускается без падений, открывается Welcome или восстановленные окна. *(MailAiApp.swift / WelcomeOrPickerScene, state restoration через WindowGroup(for: Codable) — A8.)*
- [x] Можно добавить IMAP-аккаунт (Gmail через app-password или любой стандартный IMAP-сервер) и увидеть INBOX. *(C4: OnboardingViewModel + OnboardingScene; реальный LOGIN и SELECT INBOX, затем открывается окно аккаунта.)*
- [x] Пароль сохраняется в Keychain, **не** в `Storage`, **не** в UserDefaults. *(KeychainService в `.live`, InMemorySecretsStore только в `.mock`; OnboardingViewModel стирает поле password после успеха.)*
- [x] 3-колоночный layout соответствует референсу: sidebar + list + reader с toolbar-ом. *(AccountWindowScene → NavigationSplitView; визуально проверено через `account-window-*.png` из ScreenshotSmoke.)*
- [x] Поддерживается Light/Dark тема без артефактов. *(A9: Scripts/lint-theming.sh держит код на семантических цветах; A10: ScreenshotSmoke рендерит обе темы.)*
- [x] Можно открыть 2+ окна под разные аккаунты, они изолированы (разные БД, разные сессии). *(AccountRegistry даёт отдельный `AccountSessionModel` per-id; `WindowGroup(for: Account.ID.self)` дедуплицирует окна.)*
- [x] Режим моковых данных работает без сети и показывает такой же UI. *(MOCK_DATA=1 → MockAccountDataProvider; AppShellSmoke прогоняет полный жизненный цикл.)*
- [x] При открытии письма тело загружается стримом и освобождается из памяти при закрытии. *(AccountSessionModel.open/closeSession; IMAPBodyStream; C5 IntegrationSmoke подтверждает чтение в памяти.)*
- [x] В `Storage` нет ни одного байта тела письма после закрытия сессии (проверяется тестом). *(C5: IntegrationSmoke грепит sqlite + sqlite-wal + sqlite-shm на SECRET_TOKEN и требует 0 вхождений.)*
- [x] В логах не появляются тема/отправитель/тело — только счётчики и коды ошибок. *(Проверено audit-проходом: в runtime-коде приложения логирование отсутствует; subject/from печатаются только smoke-инструментами в stderr для ручной диагностики.)*
- [x] В sidebar появляется секция «Отфильтрованные» с папками `Важное (0)` и `Неважно (0)`; клик открывает пустой список с empty state, упоминающим AI-pack. *(MockSidebarProvider → SidebarSectionKind.filtered; AccountWindowScene.aiPackEmptyState.)*
- [x] Под header'ом списка писем есть зарезервированный collapsed-слот под прогресс-бар (невидим, но layout не ломается при активации). *(UI.ClassificationProgressBar(isActive:false) → EmptyView; активный режим добавит ProgressView без перестройки.)*
- [x] В toolbar reader'а есть зарезервированный слот под иконку фоновой синхронизации (пустой в v1). *(ReaderToolbar.aiSyncSlot — disabled Button с help-строкой.)*
- [x] В Settings есть пункт «AI-классификация — недоступно» (placeholder). *(AppShell.SettingsScene → FilteredSettingsView.)*
- [x] SwiftLint — 0 warnings. Strict Concurrency — 0 warnings. *(`swiftlint --strict` в Scripts/smoke.sh — 0 violations; Strict Concurrency включена для всех SPM-модулей через swiftSettings.)*
- [x] Unit-тесты покрывают: парсинг MIME на фикстурах, upsert/fetch в `Storage`, мок `MailTransport`, ViewModel окна-аккаунта. *(XCTest-тесты под `#if canImport(XCTest)` в MailTransport/Storage/AppShell; smoke-пакет покрывает CLT-сценарии.)*

## Вне скоупа (явно)

- AI / OpenRouter — не трогаем.
- Exchange / Graph — не трогаем.
- Отправка, удаление, флажки, архив — UI есть, функциональность позже.
- FTS-поиск — поле есть, поиск не работает.
- StatusBar — нет.
- Notifications — нет.
- Настройки (preferences) — минимальные (только список аккаунтов, добавить/удалить).
- Подписывание / нотаризация — не требуется.

## Риски

| Риск | Митигация |
|---|---|
| SwiftUI `List` тормозит на 100k писем | Готовим AppKit-таблицу через `NSViewRepresentable` с самого начала, не ждём регрессии |
| IMAP-библиотеки для Swift скудные | Прототип на `MailCore2` через Obj-C bridge + fallback: своя минимальная IMAP-обёртка на `Network.framework` |
| Превью писем требует тела | Используем IMAP `PREVIEW` extension (RFC 8970) если есть, иначе — только subject + from в списке, без превью-строки |
| State restoration многооконности в SwiftUI капризная | На AppKit-стороне через `NSWindowRestoration`, если SwiftUI не тянет |
